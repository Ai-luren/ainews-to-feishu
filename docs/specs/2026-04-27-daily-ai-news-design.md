# 每日 AI 新闻推送到飞书群 —— 设计文档

**项目名**：ainews-to-feishu
**作者**：（开源版本，已移除个人信息）
**日期**：2026-04-27
**状态**：设计已批准，待实施

---

## 1. 目标

每天上午 9:00–11:00 北京时间，自动把 [橘鸦 AI 早报](https://imjuya.github.io/juya-ai-daily/) 当日新日报推送到设计团队飞书群，格式化为富文本卡片，当天只推一次。

## 2. 非目标（明确不做）

- 不做多源交叉验证（信任 juya 编辑品控）
- 不做 AI 改写/摘要（juya 已整理好，引入幻觉风险）
- 不做团队反馈按钮（需飞书企业应用，复杂度 ×5）
- 不补推历史日报
- 不做多群分发（先跑通一个群）
- 不做周报/月报汇总

## 3. 关键决策记录

| 决策 | 选择 | 理由 |
|---|---|---|
| 内容来源 | juya RSS（单一源） | 作者已做人工筛选和分类 |
| 信任策略 | 完全信任 juya | 避免 Claude 幻觉、保持简单 |
| 服务对象 | 设计团队飞书群 | — |
| 执行架构 | GitHub Actions | 云端定时，不依赖个人电脑 |
| 触发策略 | 轮询：09:01/09:31/10:01/10:31/11:01/11:31 北京时间，各跑一次（6 次） | juya 近 30 天 100% 在 10:30 前更新，轮询可在中位数下 9:30 前送达；11:31 是兜底，基本不会触发（当天已推） |
| 去重机制 | 仓库内 `state.json` 记录 "今日已推" | 简单、可审计 |
| 保护机制 | RSS 最新 pubDate ≠ 今天则跳过 | 防推陈旧内容 |
| 视觉形态 | 飞书富文本 interactive 卡片 | 团队体验与架构复杂度的甜点位 |
| 仓库 | `<你的用户名>/ainews-to-feishu` | — |
| 仓库可见性 | Private | state.json 含内部推送历史 |
| 签名校验 | 开启 | 防 webhook URL 泄露被滥用 |
| 失败告警 | 独立"运维群/私聊机器人" | 不打扰设计团队群 |
| skill 角色 | 开发/运维辅助工具，不参与运行时推送 | 推送由 GitHub Actions 独立执行 |

## 4. 架构图

```
           ┌──────────────────────────────┐
           │  juya RSS (GitHub Pages)     │
           │  imjuya.github.io/...rss.xml │
           └───────────────┬──────────────┘
                           │ 拉取
                           ▼
┌────────────────────────────────────────────┐
│  GitHub Actions (<你的用户名>/design-     │
│  team-ai-daily)                            │
│  ┌──────────────────────────────────────┐  │
│  │ cron: 6 次/天（北京时间）            │  │
│  │ 09:01 / 09:31 / 10:01 / 10:31 /      │  │
│  │ 11:01 / 11:31                        │  │
│  └───────────────┬──────────────────────┘  │
│                  ▼                         │
│  ┌──────────────────────────────────────┐  │
│  │ push.py                              │  │
│  │  1. 拉 RSS                           │  │
│  │  2. 最新 item.pubDate == 今天?       │  │
│  │  3. state.json 里今天推过了?         │  │
│  │  4. 解析 content:encoded 成卡片      │  │
│  │  5. POST 到飞书 webhook（带签名）    │  │
│  │  6. 成功 → 写 state.json 并 commit   │  │
│  └───────────────┬──────────────────────┘  │
└──────────────────┼─────────────────────────┘
                   │ HTTPS + 签名
                   ▼
    ┌─────────────────┐    ┌─────────────────┐
    │ 设计团队飞书群  │    │ 运维群/私聊     │
    │ (正常推送)      │    │ (仅错误告警)    │
    └─────────────────┘    └─────────────────┘
```

**为什么 cron 分钟用 `01/31` 而不是 `00/30`**：GitHub Actions 整点高峰排队延迟 5–15 分钟，错开 1 分钟稍减排队；同时也错开了 juya 正在发布的瞬间。

## 6. 运行时行为规则

| 情形 | 行为 |
|---|---|
| RSS 最新 pubDate = 今天（北京时区）AND state.json 里今天没推过 | **推送 + 更新 state.json + commit** |
| RSS 最新 pubDate ≠ 今天 | 跳过，打日志 "juya 尚未更新" |
| state.json 里今天已推过 | 跳过，打日志 "今日已推送" |
| 推送到飞书失败（非 200） | Actions 标红 + **不更新 state.json**（让下一轮重试） |
| 同一天连续 3 轮都失败 | 向**运维群**推一条纯文本 "⚠️ 今日推送异常 + workflow run URL" |
| 一天 6 轮都等不到 juya 更新 | 不推任何东西，Actions 成功结束；不骚扰 |
| 首次上线 | state.json 初始化为空，第一次运行如果当天有新日报则推；不补历史 |

**时区处理**：GitHub Actions 默认 UTC。`push.py` 内部统一用 `pytz` 转北京时间比较"今天"。cron 表达式按 UTC 写：`1,31 1-3 * * *` = 北京 09:01 / 09:31 / 10:01 / 10:31 / 11:01 / 11:31。

### 7.1 内容解析策略

从 RSS `content:encoded` 的 HTML 用 BeautifulSoup 提取：

- `<h2>` 或特定 class = 分组标题（"要闻" / "开发生态" / "产品应用" / "技术与洞察" / "行业动态" / "前瞻与传闻"）
- 分组下的 `<li>` 或 "标题 #编号" 结构 = 单条新闻
- B 站 / YouTube 视频链接 = 从 `<description>` 或 content 头部用正则抓
- 分类和颜色映射在 `lark_card.py` 中集中配置，**不硬编码在主逻辑里**（juya 若新增分类只改一处）

### 7.2 解析失败的降级

**永不黑屏**：如果解析失败，退化为纯文本模式 —— 标题 + 原文链接 + "解析异常，点击看原文" —— 仍然推送。并且在运维群发一条 "⚠️ 今日内容解析降级" 告警。

## 8. 安全与密钥

| 密钥 | 存放位置 | 用途 |
|---|---|---|
| `LARK_WEBHOOK_URL` | GitHub Secrets | 设计团队群 webhook |
| `LARK_WEBHOOK_SECRET` | GitHub Secrets | 设计团队群签名密钥 |
| `LARK_OPS_WEBHOOK_URL` | GitHub Secrets | 运维群/私聊 webhook |
| `LARK_OPS_WEBHOOK_SECRET` | GitHub Secrets | 运维群签名密钥 |

**绝对禁止**：这 4 个值出现在代码、README、commit message、日志输出中。

**签名算法**：按飞书官方文档 HMAC-SHA256 → base64，代码中实现一个 `lark_sign(secret, timestamp) -> str` 工具函数。

## 9. skill 的角色

skill 名：`daily-ai-news`，路径 `~/.claude/skills/daily-ai-news/SKILL.md`。

**skill 不做推送**。skill 做的事：

1. **`/daily-ai-news setup`** — 首次搭建向导
   - 创建 GitHub 仓库（用 `gh repo create`）
   - 写好 push.py / lark_card.py / workflow
   - 指引用户去飞书拿 2 对 webhook + secret
   - 用 `gh secret set` 把 4 个密钥存到 GitHub Secrets
   - 手动触发一次 workflow 验证

2. **`/daily-ai-news test`** — 强制推一次到**运维群**（不是正式群），无视今天是否推过
3. **`/daily-ai-news status`** — 查看最近 7 天的推送记录 / Actions 执行历史
4. **`/daily-ai-news change-time`** — 改 cron 表达式
5. **`/daily-ai-news change-group`** — 换 webhook

**排障模式**：当用户说"早报没到"，Claude 按 skill 的诊断流程查 Actions 日志 → 测 RSS → 测 webhook → 定位。

## 10. 成功指标

- **上线 7 天内**：5/7 天按时推送成功（允许 juya 自身断更 2 天）
- **卡片可读性**：团队成员扫一眼能找到自己关心的分类，点击链接能正确跳转
- **零泄露**：4 个密钥 100% 留在 GitHub Secrets，不进任何 commit

## 12. 风险与应对

| 风险 | 概率 | 影响 | 应对 |
|---|---|---|---|
| juya 断更 / RSS 挂掉 | 中 | 高 | 跳过推送；连续 3 天无更新向运维群告警 |
| juya 改变 HTML 结构 | 低 | 中 | 解析失败自动降级为纯文本 + 运维群告警 |
| GitHub Actions 延迟超 30 分钟 | 低 | 低 | 6 次轮询覆盖 2.5 小时窗口 |
| 飞书 webhook 失效 | 极低 | 高 | 失败不更新 state.json，下轮重试；连续失败告警 |
| 密钥泄露 | 极低 | 高 | 签名校验 + Private 仓库 + 仅 Secrets 存储 |
